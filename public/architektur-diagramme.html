01 · Strukturdiagramm

Klassendiagramm der drei Kontrakte

Die Architektur trennt Geschäftslogik (PalindromePay), deterministische Wallet-Erzeugung (Factory, CREATE2) und Verwahrung der Mittel (ein Wallet pro Deal). Das Wallet vertraut dem Escrow-Kontrakt nicht blind: Es liest den Deal-Zustand zurück und verlangt bei der Auszahlung eigene, outcome-gebundene EIP-712-Signaturen von 2 der 3 Beteiligten.

%%{init: {'theme':'neutral'}}%%
classDiagram
    direction LR

    class PalindromePay {
        +address FEE_RECEIVER «immutable»
        +IWalletFactory WALLET_FACTORY «immutable»
        +uint256 nextEscrowId
        -mapping escrows : id → EscrowDeal
        -mapping usedSignatures
        -mapping nonceBitmap
        +uint256 disputeStatus «Bitmap Beweise»
        +createEscrow()
        +createEscrowAndDeposit()
        +deposit()
        +acceptEscrow(sellerSig)
        +confirmDelivery(buyerSig)
        +confirmDeliverySigned(...)
        +requestCancel()
        +cancelByTimeout()
        +autoRelease()
        +startDispute()
        +startDisputeSigned(...)
        +submitDisputeMessage(ipfs)
        +submitArbiterDecision(resolution, arbiterSig)
        +refundAfterDisputeTimeout(buyerSig)
        +setArbiterSigned(...)
        +getEscrow() EscrowDeal
    }

    class PalindromePayWalletFactory {
        +deploy(escrowId) address
        +predictWalletAddress(escrow, escrowId) address
    }

    class PalindromePayWallet {
        +address ESCROW_CONTRACT «immutable»
        +uint256 ESCROW_ID «immutable»
        +bool withdrawn
        +withdraw()
        +getValidSignatureCount(outcome)
        +isSignatureValid(participant, outcome)
        +getAuthorizationDigest(participant, outcome)
        -_isAuthorized()
        -_computeFeeAndNet()
    }

    class EscrowDeal {
        «struct»
        +address token
        +address buyer
        +address seller
        +address arbiter
        +address wallet
        +uint256 amount
        +uint256 maturityTime
        +State state
        +uint16 arbiterFeeBps
        +bytes sellerWalletSig
        +bytes buyerWalletSig
        +bytes arbiterWalletSig
    }

    class State {
        «enum»
        AWAITING_PAYMENT
        AWAITING_DELIVERY
        DISPUTED
        COMPLETE
        REFUNDED
        CANCELED
    }

    class Role {
        «enum»
        None
        Buyer
        Seller
        Arbiter
    }

    PalindromePay "1" o-- "n" EscrowDeal : verwaltet
    EscrowDeal --> State : state
    PalindromePay ..> Role : Ereignisse
    PalindromePay --> PalindromePayWalletFactory : WALLET_FACTORY
    PalindromePayWalletFactory ..> PalindromePayWallet : CREATE2-Deploy pro Deal
    PalindromePayWallet --> PalindromePay : liest getEscrow(), FEE_RECEIVER()

Die Signaturen im EscrowDeal werden vom Escrow-Kontrakt nur gesammelt — kryptografisch geprüft werden sie erneut im Wallet beim withdraw(). Jede Signatur bindet sich per PayoutAuthorization-Typehash an genau ein Outcome (COMPLETE = 3, REFUNDED = 4, CANCELED = 5).

02 · Verhaltensdiagramm

Zustandsdiagramm des Escrow-Lebenszyklus

Sechs Zustände, drei davon terminal. Terminale Zustände geben die Mittel frei — wer sie erhält, bestimmt der Zustand selbst: COMPLETE zahlt an den Seller (abzüglich 1 % Plattform-Fee, bei Arbiter-Entscheidung zusätzlich Arbiter-Fee), REFUNDED und CANCELED erstatten dem Buyer den vollen Betrag.

%%{init: {'theme':'neutral'}}%%
stateDiagram-v2
    [*] --> AWAITING_PAYMENT : createEscrow()
    [*] --> AWAITING_DELIVERY : createEscrowAndDeposit()

    AWAITING_PAYMENT --> AWAITING_DELIVERY : deposit() — Buyer
    AWAITING_PAYMENT --> CANCELED : requestCancel() beidseitig / Timeout

    AWAITING_DELIVERY --> COMPLETE : confirmDelivery(buyerSig) — Buyer
    AWAITING_DELIVERY --> COMPLETE : autoRelease() — Seller, nach maturityTime,\nSeller-Sig vorhanden, kein Dispute/Cancel
    AWAITING_DELIVERY --> CANCELED : requestCancel() beidseitig /\ncancelByTimeout()
    AWAITING_DELIVERY --> DISPUTED : startDispute() — Buyer oder Seller

    DISPUTED --> COMPLETE : submitArbiterDecision(COMPLETE)\nBeweise beidseitig oder 30 d + 1 h
    DISPUTED --> REFUNDED : submitArbiterDecision(REFUNDED)
    DISPUTED --> REFUNDED : refundAfterDisputeTimeout()\nBuyer, nach 30 d + 1 h ohne Entscheidung

    COMPLETE --> [*] : withdraw() → Seller − 1 % Fee (± Arbiter-Fee)
    REFUNDED --> [*] : withdraw() → Buyer (voll)
    CANCELED --> [*] : withdraw() → Buyer (voll)

Der Arbiter darf erst entscheiden, wenn beide Parteien Beweise eingereicht haben (disputeStatus-Bitmap) — oder ersatzweise nach Ablauf von DISPUTE_LONG_TIMEOUT (30 Tage) + TIMEOUT_BUFFER (1 Stunde). Entscheidet er gar nicht, kann der Buyer sich nach derselben Frist selbst erstatten.

03 · Aktivitätsdiagramm

Escrow-Ablauf: Happy Path und Dispute-Pfad

Der Normalfall kommt ohne Arbiter aus — er wird nur im Streitfall aktiv (und nur dann fällt seine Fee an). Die Wallet-Adresse ist per CREATE2 schon vor dem Deposit deterministisch bekannt.

%%{init: {'theme':'neutral'}}%%
flowchart TD
    A([Buyer/Seller einigen sich auf Deal]) --> B[createEscrow bzw. createEscrowAndDeposit\nToken, Betrag, Frist, optional Arbiter + Fee]
    B --> C[Wallet-Adresse via CREATE2\ndeterministisch erzeugt]
    C --> D[Buyer: deposit\nToken fließen in das Deal-Wallet]
    D --> E[Seller: acceptEscrow\nsigniert Outcome COMPLETE]
    E --> F{Lieferung /\nLeistung in Ordnung?}

    F -- ja --> G[Buyer: confirmDelivery\nsigniert Outcome COMPLETE]
    G --> H[[State → COMPLETE]]

    F -- Frist verstrichen,\nBuyer reagiert nicht --> I[Seller: autoRelease\nnach maturityTime]
    I --> H

    F -- nein --> J[startDispute\nBuyer oder Seller]
    J --> K[Beide Parteien: submitDisputeMessage\nBeweise als IPFS-Hash]
    K --> L{Arbiter entscheidet\ninnerhalb 30 Tagen?}
    L -- ja --> M[submitArbiterDecision\nsigniert exakt das Urteil]
    M --> N{Urteil}
    N -- für Seller --> H
    N -- für Buyer --> O[[State → REFUNDED]]
    L -- nein --> P[Buyer: refundAfterDisputeTimeout\nnach 30 d + 1 h]
    P --> O

    F -- beide wollen abbrechen --> Q[requestCancel von Buyer und Seller]
    Q --> R[[State → CANCELED]]

    H --> S[withdraw am Wallet\n2-of-3-Signaturprüfung]
    O --> S
    R --> S
    S --> T([Auszahlung: COMPLETE → Seller − Fees,\nsonst → Buyer voll])

04 · Sequenzdiagramm

Outcome-gebundene 2-of-3-Auszahlung

Der Kern des Sicherheitsmodells: Kein einzelner Teilnehmer — auch nicht der Escrow-Kontrakt selbst — kann Mittel bewegen. Das Wallet prüft beim withdraw() selbstständig, dass mindestens zwei Teilnehmer genau das eingetretene Outcome signiert haben.

%%{init: {'theme':'neutral'}}%%
sequenceDiagram
    autonumber
    actor B as Buyer
    actor S as Seller
    participant E as PalindromePay
    participant W as Deal-Wallet
    participant T as ERC20-Token

    B->>E: deposit(escrowId)
    E->>T: transferFrom(Buyer → Wallet)
    Note over E: State = AWAITING_DELIVERY

    S->>E: acceptEscrow(sellerSig)
    Note right of E: EIP-712-Sig, Outcome = COMPLETE,\nwird im Deal gespeichert

    B->>E: confirmDelivery(buyerSig)
    Note right of E: EIP-712-Sig, Outcome = COMPLETE
    E->>E: _setState(COMPLETE), Payout vorgemerkt

    S->>W: withdraw()
    W->>E: getEscrow(ESCROW_ID)
    E-->>W: EscrowDeal (state, Beteiligte, Sigs)
    W->>W: Prüfe ≥ 2 gültige Sigs\nfür Outcome COMPLETE
    W->>T: transfer(1 % → FEE_RECEIVER)
    W->>T: transfer(Rest → Seller)
    Note over W: withdrawn = true,\nWallet muss leer sein

Im Dispute-Fall liefert stattdessen der Arbiter die zweite Signatur — submitArbiterDecision bindet sie an genau das gefällte Urteil. Eine COMPLETE-Signatur kann niemals eine REFUNDED-Auszahlung autorisieren und umgekehrt; Replay wird über usedSignatures und Nonce-Bitmaps verhindert.

05 · Aktivitätsdiagramm

Deployment via Trezor + Frame

Der gehärtete Ablauf aus scripts/deploy.ts --trezor: Jede Guard bricht vor der ersten Transaktion ab, signiert wird ausschließlich auf dem Gerät.

%%{init: {'theme':'neutral'}}%%
flowchart TD
    A([Start: deploy.ts <netz> --trezor]) --> B{Frame erreichbar\nauf 127.0.0.1:1248?}
    B -- nein --> X1[[Abbruch: Frame starten,\nTrezor entsperren]]
    B -- ja --> C{Frame-Chain ==\nZiel-Chain-ID?}
    C -- nein --> X2[[Abbruch: Chain-Badge\nin Frame umstellen]]
    C -- ja --> D[Deployer = erster Frame-Account\nFee Receiver aus .env]
    D --> E[Gas-Schätzung aller Deploys]
    E --> F{Balance ≥\nKosten + 20 %?}
    F -- nein --> X3[[Abbruch: Insufficient balance]]
    F -- ja --> G{Bestätigung „yes"\noder --yes-Flag?}
    G -- nein --> X4[[Abbruch durch Operator]]
    G -- ja --> H[Deploy Factory\nBestätigung auf Trezor]
    H --> I[Deploy PalindromePay\nfeeReceiver + Factory-Adresse]
    I --> J[Warten: 5 Confirmations Mainnet\n2 Testnet]
    J --> K{Sanity-Check:\nFEE_RECEIVER und\nWALLET_FACTORY on-chain korrekt?}
    K -- nein --> X5[[Fehler + partial-Record]]
    K -- ja --> L[deployments/<netz>.json schreiben]
    L --> M[Etherscan-Verifikation]
    M --> N([Fertig — Record committen])

Auf Testnetzen wird zusätzlich der Test-USDT als erste Transaktion deployt; auf Mainnet entfällt das per --escrow-only. Schlägt nur die Verifikation fehl: nicht neu deployen, sondern den ausgegebenen --verify-only-Befehl nutzen.
